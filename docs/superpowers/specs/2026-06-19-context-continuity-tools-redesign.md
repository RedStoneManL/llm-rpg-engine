# 叙事上下文 / 剧情连续性 / 工具化 重构 — 设计 spec

> 这是在活世界层 A–E(744→738 tests)之上的**下一个子项目**。源于两个发现:(1) cascade 横向 spread 的判断被错放在廉价模型(它没故事上下文,一律不蔓延);(2) 更根本——"剧情走到哪"在现架构里是**隐式**的、最易断的一环。讨论后收敛出一套重构。

## 0. 动机(现状的洞)

- **状态一致性**(谁在哪/什么是真的/谁知道什么):事件溯源 + 严格门 + 投影,**硬,不漂**。
- **剧情连续性**("当前目标/未决伏笔/张力/主线到第几拍"):**没有一等公民**追踪它,DM 靠 recall 捞零碎 + 自己脑补 → 最易断。
- **cascade 触发**:被错改成"harness 自读事件 + 廉价模型逐节点猜 spread" → 横向永不触发。
- **叙事模型看不到全局**:只能用塞进 prompt 的上下文,不能按需查地图/人物/组织 → 也是乙身份碎片化的根源。

## 1. 组织原则:推(PUSH) vs 拉(PULL);读用工具、写走闸门

- **PUSH(强制,每回合必看)= 延续性命脉**:recap(分层) + 故事线账本 + 当前场景态 + §9 视野。**不给模型选择权**(不靠 recall 捞)——这些断了剧情就崩。
- **PULL(按需,工具)= 参照**:地图/人物/组织/记忆,查得到就行,丢了不致命。
- **keystone:工具只读**(叙事模型的"感官",看遍世界但改不了状态);**一切写入仍走严格校验门**(turn-commit)。"autonomy within deterministic guardrails"。

## 2. 组件

### 2.1 cascade 改为 `world` 段驱动(取代自触发 + 廉价猜 spread)
- 叙事模型在 turn-commit 产 `world` 段(像 knowledge 段那样暴露给甲/丙):
  ```
  world: [{ "areas": [地点id,...],   // 叙事模型点名所有受影响区域(任意位置,不限邻居)
            "level": 1|2|3,
            "summary": "一句话事件" }]  // 廉价模型靠它知道在传播什么
  ```
- 一个 system 认领 `world` 段 → world_change 事件。cascade 退化成**纯执行器**:对每个 `area` 纵向下沉(廉价模型填后果细节),**次级 spread 最多一层**(廉价模型在每个 area 自行点名再外扩一格、不再深入)。
- 横向/纵向统一成"声明的地点挨个渲染 + 廉价至多补一圈"。"是否/往哪蔓延"=叙事模型(有故事脑)定;"烧成啥样 + 末端再扩一格"=廉价舰队定。
- **保留**现有并行 fan-out / 懒延迟队列 / 合并同区(执行机制),只换触发源。

### 2.2 recap(递归分层,PUSH)
- 最近 N 章原文 + 更老滚成摘要 + 再老摘要的摘要。近有细节、远有兜底(借上下文压缩的精神)。
- 后台 digest 舰队维护(章落出窗口就压缩);每回合**强制注入**给叙事模型。
- 这是设计 §15"两层+递归压缩"和延后的"双视图编年"的落地上层。

### 2.3 故事线账本(PUSH)
- 结构化 `{线id, 状态:活跃|休眠|已结, 一句话摘要, 上次推进章}`,每回合**全量推**,不让模型选择性观看。
- 跟暗骰 director 的 threads **不是一回事**(那是隐藏节奏种子;这是玩家主线明账)。

### 2.4 工具(PULL,只读)
- `map.query`(地点 contained_by/adjacent_to/level/path)、`characters.query`(canonical id/sketch/goal/位置/关系/arc)、`factions.query`(成员/阶/关系)、`recall(query)`(语义召回,已有)。全是把 FactGraph + 现有 system 包成只读查询,无新数据。
- **战争迷雾(关键)**:按查询意图分**两套物理入口**——
  - **POV 工具**("主角此刻能感知/回忆什么"):只返回主角/被问 NPC 实际知道的(复用 §9 knows 事实) → 源头防泄。
  - **DM 工具**("作者需要的世界真相":cascade 范围、跨区一致性):返回真相,但**钉上 `protagonist_knows:false`**,且只喂给作者/结构环节、**不直接进 narration 上下文**。
  - 迷雾不靠模型自觉,靠入口分流。

### 2.5 provider 工具循环
- GLM/质谱 OpenAI 式 function-calling:模型返回 tool_calls → 执行 → 回灌 → 续,直到出最终 commit。
- 形态 **research-then-write**(先查够、再一次写完 narration+结构),跟严格门/agent-loop 兼容。

## 4. 分阶段(每阶段做完→真模型看效果→提交;先出最能看到效果的)

- **P1 · cascade world-段驱动**(最具体、最完成 C2、可演示):`world` 段 owner + 暴露进甲/丙 prompt + cascade 读它而非自触发 + 次级一层 + 廉价模型节点加"再扩一格"决策。**横向终于能由叙事模型驱动地触发。**
- **P2 · 剧情连续性**(用户最在意的洞):recap 分层 + 故事线账本 + 强制推注入。digest 舰队扩成维护 recap/故事线。
- **P3 · 工具化**(最大、碰 provider):provider 工具循环 + map/characters/factions 工具 + POV/DM 双入口 + 推/拉上下文重排。**P3 一上,乙的老毛病彻底没意义、甲/丙可能合一。**
- 每阶段:TDD plan → subagent 实现+评审 → 真模型 `/tmp/verify_*.py` 看效果 → 提交。**离线可确定性测试是命脉**(P3 的工具循环要能伪造 tool-call 序列,别把可测性搞脆)。

## 5. 不变量(沿用 + 新增)
- 写入边界唯一 = turn-commit 严格门;工具只读不可绕过它。
- 投影永不崩;harness 自生成事件(world_change/recap/storyline 维护)走轻量 referential 校验、不走完整修复。
- 推集每回合 token 更重是**故意的取舍**(延续性值这个钱);分层 recap 保证有界、不无限涨。
